Is there an existing open-source "time machine" so you don't rebuild the wheel? Yes — several (dbt snapshot, Snowflake Streams, XTDB, Datomic, Dolt, Hudi). They solve ~90% of FluxState. But none owns FluxState's exact niche: a single-table/view, engine-portable (Polars-on-anything), cell-level change historian whose output is a ready-to-render per-cell timeline — and that runs with zero DDL rights off Snowflake entirely. That sliver is real and defensible. The catch: the current implementation is a scaling bomb, so "keep" only wins if you fix the storage model.

Your single-table premise holds up. Pointing at one table/view (not the whole DB) is genuinely surgical — you pay temporal-state cost only for the surface you care about. Whole-DB engines (XTDB, Datomic, Snowflake Time Travel) version everything. That selectivity is a point in FluxState's favour, not a misunderstanding.

Why not just drop it — the honest case for each rival

  • dbt snapshot (SCD2) — battle-tested, near-zero new code, great for population compliance queries. But row-versioned, Snowflake-resident, no per-cell timeline object, useless off-warehouse.
  • Snowflake Streams + Tasks — cheapest possible CDC, only processes real changes. But hard vendor lock + needs CREATE STREAM / EXECUTE TASK rights you may not have.
  • XTDB / Datomic — true bitemporal time-travel SQL. But whole-DB engines — not a single-table drop-in, big adoption tax.
  • Oxen diff --keys / daff — ready-made cell-level diff. Borrow the idea, not the dependency (see infusion call below).

30-second decision tree

dataset >10M rows OR high-velocity? └─ yes → DROP (Streams / dbt) └─ no → must run on DuckDB/CSV/Postgres? └─ yes → KEEP (the portable niche) └─ no → main query = population-wide compliance / whole-table PIT? └─ yes → have DDL rights? └─ yes → DROP (SCD2) └─ no → KEEP (pivot) └─ no → KEEP (timeline-UI win)

KEEP overrides even at scale if: no CREATE-STREAM rights (shadow-IT analyst), schema drifts weekly (ragged source), or the source has no clean primary keys.

When to revisit & drop later

The one query that flips the call to DROP:

"Show me the state of all 50,000 patients as of Jan 12, 2023" — a whole-table point-in-time reconstruction.
• Once a year → keep FluxState (checkpoint files cover it).
• Every Monday → drop to dbt-snapshot / Streams; SQL engines are built for that join, Polars will hit a RAM ceiling.

Other drop triggers: legal needs cryptographic chain-of-custody (who changed it, not just what); data crosses into millions of rows with daily full-table audits.

Infusion: Oxen / daff?

Do not hard-depend on either. Re-implement the Oxen-style keyed cell-diff in ~20 lines of Polars (already a dependency) — that alone deletes the O(R²) bug. Keep daff / Oxen as an optional "pretty diff" renderer for human audit review only, never the core engine. The IO/CLI tax isn't worth it for a single-table batch job.

Improvements worth adding

  • Storage Adapter interface (LocalFS / S3 / Snowflake-stage) so "engine-agnostic" is true — today it secretly assumes a filesystem.
  • Checkpoint compaction every N runs → keeps whole-table time-travel from going O(N).
  • Row-hash skip: a tiny (id, row_hash) summary lets you skip unchanged rows instantly at scale.
  • generate_audit_bundle(id) → self-contained offline HTML for a compliance officer.

Bottom line

Don't drop it, and don't ship it as-is. The niche is real — keep FluxState as the portable, zero-DDL, single-table cell historian — but spend ~5–6 focused days converting it from a "rewrite-the-whole-mirror" snapshot tool into an append-only change-log with on-demand reconstruction. That single move neutralises every HIGH-severity bug at once and makes the "engine-agnostic watcher" pitch actually true. Re-evaluate the moment whole-table, population-wide point-in-time audits become a weekly need — that's the day dbt-snapshot wins.
